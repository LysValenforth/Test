TOPIC 01
Priority Scheduling
PREEMPTIVE · NON-PREEMPTIVE · STARVATION · TIPS & TRICKS
NON-PREEMPTIVE

No Interruptions

Once a process starts, it runs to completion. A higher-priority arrival waits until the CPU is free.

PREEMPTIVE

Higher Priority Kicks Out

A new process with higher priority (lower number) immediately takes the CPU. The interrupted process resumes later from where it stopped.

STARVATION

Low-Priority Processes Wait Forever

If high-priority jobs keep arriving, low-priority ones may never execute. This is the main disadvantage of priority scheduling.

PRIORITY RULE

Lower Number = Higher Priority

Priority 1 beats Priority 2. The CPU always goes to the process with the smallest priority number currently ready.

⚡ Priority Assignment Methods

The processor manager can assign priorities based on any of these factors:

FactorRule
Memory requirementsLarge memory → Lower priority
Peripheral devices neededMany peripherals → Lower priority
Total CPU timeLong CPU cycle → Lower priority

🧠 How to Solve — Non-Preemptive Priority

1

At time 0 (or when CPU is free) — look at all processes that have arrived. Pick the one with the lowest priority number.

2

Run it to completion. Don't interrupt for any reason, even if a higher-priority process arrives mid-execution.

3

When it finishes — from all currently-arrived processes, again pick the lowest priority number.

4

If CPU is idle (no process has arrived yet) — fast-forward time to the next arrival.

5

Calculate: Completion Time (CT) → Turnaround = CT − Arrival → Waiting = Turnaround − Burst.

💡 Trick: Draw the Gantt chart first, fill the table after. It's much easier to spot idle gaps and process order visually.

⚡ How to Solve — Preemptive Priority

1

Track remaining burst time for every process — subtract however long it ran before being interrupted.

2

At every arrival event — pause and compare the new process's priority with the current runner. If lower number → preempt.

3

Only re-check at arrival times, not every millisecond. Events occur only when a new process arrives or the current one finishes.

4

Track remaining burst in slash notation: e.g., P2: 3/1/0 means started with 3, had 1 left after preemption, then finished.

5

Calculate same formulas: CT → Turnaround = CT − Arrival → Waiting = Turnaround − Burst.

💡 Trick: Make a mini "event table" listing every arrival time. At each event, write down who is currently running and whether the newcomer can kick them out.

TOPIC 02
Round Robin
TIME QUANTUM · PREEMPTIVE · FAIR · STARVATION-FREE
TIME QUANTUM

Equal Time Slices

Each process gets a fixed time quantum (e.g., 15ms, 20ms). After the quantum expires, the process goes to the back of the queue.

ALWAYS PREEMPTIVE

Timer Kicks Everyone Out

Even if a process hasn't finished, it's preempted when the quantum expires. The OS then picks the next process in the ready queue.

NO STARVATION

Everyone Gets a Turn

Unlike priority scheduling, every process cycles through the queue. No process can monopolize the CPU indefinitely.

PCB SAVES STATE

Process Control Block

When a process is preempted, its state (registers, remaining burst) is saved in its PCB so it can resume correctly next round.

🔄 How to Solve Round Robin — Step by Step

1

Make a remaining-burst column. Start with each process's full burst time (e.g., 70/50/30… as you subtract each round).

2

Maintain a ready queue. Initially, add processes in arrival order. When a process arrives during a quantum, it joins the queue at that point.

3

Each round: Run the front-of-queue process for min(remaining, quantum). If it finishes early, it's done; otherwise subtract quantum and move it to the back.

4

New arrivals during a quantum join the queue before the just-preempted process goes back. Arrival order determines position.

5

Calculate: Completion Time (when remaining = 0) → Turnaround = CT − Arrival → Waiting = Turnaround − Burst.

💡 Key Trick — Quantum vs Remaining: If remaining burst < quantum, the process uses only the remaining time and finishes. Don't blindly subtract the full quantum!
⚠️ Quantum Size Warning:
Too large → degenerates into FCFS (processes finish before quantum).
Too small → too much context switching, system slows down.
Ideal: quantum long enough for 80% of CPU cycles to complete.

Formula Sheet
ALL KEY FORMULAS FOR CPU SCHEDULING
TURNAROUND TIME
Total Time in System
TAT = CT − Arrival Time
CT = when process fully completes
Arrival = when it entered the system
WAITING TIME
Time Spent Waiting
WT = TAT − Burst Time
TAT = turnaround time
Burst = actual CPU time needed
AVERAGE TAT
Mean Turnaround
Avg TAT = Σ TAT / n
Sum all turnaround times
Divide by number of processes
AVERAGE WAITING
Mean Waiting Time
Avg WT = Σ WT / n
Sum all waiting times
Divide by number of processes
ROUND ROBIN — RUN TIME
Each Slice Duration
Run = min(Remaining, Quantum)
If remaining < quantum
→ process finishes early
PREEMPTIVE PRIORITY
Remaining Burst After Interrupt
Remaining = Burst − Time Run
Track per-interrupt in slash notation
e.g., P2: 3 / 1 / 0
CPU IDLE TIME
Gap Before Next Arrival
Idle = Next Arrival − Current Time
Occurs when ready queue is empty
and next process hasn't arrived yet
COMPLETION TIME
When Process Finishes
CT = Start Time + Burst Time
Or last time it ran + remaining burst
in preemptive/RR cases
